# Task: 1-A-3 — QFaultIRModule with two-level support

## Description

Define `QFaultIRModule` — the central data structure that holds the entire quantum circuit at either the logical or physical level. This is the key Stage 1 design decision (ADR-0001): one module, two levels, discriminated union of instructions.

## Acceptance Criteria

- [ ] `enum class IRLevel { LOGICAL, PHYSICAL };` defined in `include/qfault/ir/IRLevel.hpp`
- [ ] `using Instruction = std::variant<LogicalGate, PatchOp>;` defined
- [ ] `struct QFaultIRModule` fields: `std::string name`, `IRLevel level`, `std::vector<LogicalQubit> qubits`, `std::vector<Instruction> instructions`
- [ ] `QFaultIRModule::assertLevel(IRLevel required)` throws/asserts if `level != required`
- [ ] `tests/unit/test_QFaultIRModule.cpp`: tests for LOGICAL-level module (append LogicalGates, assertLevel passes), PHYSICAL-level module (append PatchOps), and assertLevel failure on wrong level
- [ ] `clang-tidy` clean; all `enum class`; no raw `new`/`delete`

## Technical Details

- `assertLevel` should use `assert()` in debug builds and throw `std::logic_error` in release — or use a single `if (...) throw std::logic_error(...)` that always fires
- Do NOT put `std::variant` inside `LogicalGate` or `PatchOp` payload sub-types (compile-time blowup — see ADR-0001 and CHANGELOG.md)
- Use `std::visit(overload{...}, instruction)` pattern for any dispatch over instructions
